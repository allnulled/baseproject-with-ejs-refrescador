# POOF: Programación Orientada a Objetos Funcionales

Es POO pero usas clases a modo de funciones.

## Reglas

### Regla 1. Todo clases

Usas clases para todo, todo lo encapsulas en clases, como Java.

### Regla 2. Una clase por proceso y subproceso

- Cada vez que tienes un proceso o subproceso, creas otra clase.
- Y descompones sus funcionalidades en propiedades locales y métodos prototipo

La equivalencia entre OOP y FunctionalP:

```js
Modo_oop_minimalista: {
    class Proceso {
        constructor() {
            static create(...args) { return new this(...args); }
            constructor(settings) {
                this.settings = settings;
            }
            usage(specifications) {
                const opciones = Object.assign({}, this.parametro, especificaciones);
                // Lógica de la función
            }
        }
    }
    Proceso.create({ settings: "iniciales" }).usage({ parametros: "esporádicos" });
}
Modo_fp: {
    const Proceso = function(parametros, especificaciones) {
        // Lógica de la función
    };
    Proceso({ settings: "iniciales" }, { parametros: "especificos" });
}
Modo_oop_avanzado: {
    class ParametroDeProceso {}
    class ResultadoDeProceso {
        constructor(opciones) {
            // Lógica de la función
        }
    }
    class Proceso {
        static create(...args) { return new this(...args); }
        constructor(parametro) {
            this.parametro = parametro;
        }
        usage(especificaciones) {
            const opciones = new ParametroDeProceso(Object.assign({}, this.parametro, especificaciones));
            return new ResultadoDeProceso(opciones);
        }
    }
    Proceso.create({ settings: "iniciales" }).usage({ parametros: "esporádicos" });
}
```

### Regla 3. Aplanar métodos en cada clase

- Los métodos de clase están a 2 niveles
   - `static` o
   - `prototype`

### Regla 4. Diferenciar métodos públicos y privados

- Los privados van con `_` de prefijo
- Los privados sería para que no se vieran desde el usuario final
   - No usar sintaxis de campos privados de ES6
      - porque perderías accesibilidad
      - aunque sean métodos secundarios, no sabes cuándo vas a requerirlo
         - ni que sea a modo temporal, hasta que definas otra clase heredada o lo que sea
   - Los privados son métodos que:
      - están ahí
      - son accesibles pero
      - podríamos considerar como si no estuvieran
         - para un uso superficial de la API
         - para un uso más profundo, puedes accederlos con `_` igual que los públicos normal

### Regla 5. Usar la sintaxis de método de ES6

- Los métodos `prototype` solo se verán si usas esta sintaxis:

```js
class Whatever {
    // No se ve en el prototype:
    notPrototype = function() {}
    // Sí se ve en el prototype:
    yesPrototype() {}
}
```

### Ventajas

- Reaprovechas todos los métodos de todas las clases
   - porque puedes acceder a su prototype para reaprovechar los métodos dinámicos de la clase
   - eso sí, tienes que llamar a `Function.prototype.call` y proporcionar el `this` para que los métodos no fallen.
- Reaprovechas todos los métodos de todas las clases
